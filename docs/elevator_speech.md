## Speech Script

### 1. 문제
여러분, 건강 상태를 정확히 알고 관리하고 계신가요?
최근 5년간 20~30대의 당뇨병 유병률은 38%, 고혈압은 28%나 증가했습니다.
하지만 기존 건강관리 앱은 번거로운 수동 입력과 획일적인 조언에 그쳐, 바쁜 청년들이 꾸준히 관리하기 어려운 한계가 있습니다.

### 2. 해결 아이디어
On-Care는 사진 한 장으로 식단을 기록·분석하고, 누적 건강 데이터를 학습한 AI 코치가 개인 맞춤 피드백을 제공합니다. 
또한 식단, 운동, 헬스장, 일정 관리까지 하나의 앱에서 통합적으로 해결합니다.

### 3. 기술 / 구현
식단 인식은 2단계 파이프라인으로 동작합니다. 
먼저 YOLOv8이 음식 여부를 판별해 불필요한 API 호출을 차단하고, Gemini Vision API가 음식 종류와 섭취량을 분석해 영양 정보를 자동 기록합니다.
AI 코칭은 RAG 기반으로 구현하고 개인 건강 데이터를 Pinecone에 저장하여, 사용자 기록을 반영한 맞춤형 답변을 제공합니다.
시스템은 FastAPI, Flutter 기반으로 구축됩니다.

### 4. MVP / 배포
현재 Gemini API 연동과 프롬프트 설계를 완료했습니다. 
MVP에서는 사진 기반 식단 기록, AI 코칭, 운동 루틴 및 캘린더 연동 기능을 제공합니다. 
앱은 iOS와 Android 동시 출시를 목표로 개발 중이며, 서비스는 Docker 환경에서 운영할 예정입니다.

### 5. 차별성
기존 앱들은 기록과 코칭이 분리되거나, 오프라인 운동 환경과 단절되어 있습니다. 저희 서비스, On-Care는 내 누적 기록을 실시간 참조하는 RAG 기반 AI 코치와 헬스장 연동을 하나로 묶어, 기록에서 실제 행동 변화까지 연결합니다.

## 문제 (Problem)
 
### Pain Point
 
**① 기록해도 행동이 바뀌지 않는다**
기존 앱은 상세 기록을 요구하지만, 기록 → 분석 → 행동 변화로 이어지는 흐름이 끊겨 있다. 사용자는 앱을 이탈하거나 O/X 체크 수준으로 사용을 축소한다.
 
**② 기록 자체가 너무 번거롭다**
반복되는 식단임에도 매번 텍스트 검색·바코드 입력이 필요하다. 사진 인식을 제공하는 앱도 있으나 복합 요리 정확도가 낮아 수동 보정이 요구된다.
 
**③ 내 질환 기준의 조언이 없다**
기존 헬스케어 앱 대부분이 칼로리 알림 수준에 머문다. 만성질환 위험군에게 필요한 것은 나트륨·혈당 기준 경고와 대체 식품 추천이다.
 
**④ 다이어트 앱 구조가 질환 관리에 맞지 않는다**
기존 앱은 "얼마나 먹었는가" 중심으로 설계되어 있다. 만성질환 위험군의 핵심 관리 포인트는 "무엇을 피해야 하는가"이다.
 
**⑤ 식단·운동·상담 서비스가 파편화되어 있다**
경쟁 제품 모두 오프라인 헬스장·트레이너 연결 기능이 없거나 부족하다. 사용자는 정보를 얻기 위해 앱 밖으로 이탈해야 한다.
 
### 문제 한 줄 요약
 
기존 헬스케어 앱은 기록 중심·비개인화·파편화된 구조로 인해, 만성질환 위험군의 지속적 건강 관리와 실제 행동 변화를 만들어내지 못하고 있다.

## 해결 아이디어 (Solution)
 
**On-Care** — 고혈압·당뇨 위험군을 위한 Vision AI + RAG 기반 통합 헬스케어 앱
 
- 사진 한 장으로 식단 자동 인식 및 영양 분석
- 질환 프로필 + 누적 식단 이력을 아는 AI 코치가 맞춤 경고·추천 제공
- 식단·운동·헬스장·일정을 단일 플랫폼에서 통합
핵심 가치: "기록 도구"가 아닌 "행동 변화를 만드는 앱"

## 기술 / 구현 (Tech Stack)
 
| 영역 | 기술 |
|---|---|
| 모바일 | Flutter (iOS·Android 동시 지원) |
| 백엔드 | FastAPI / Docker / AWS EC2 |
| 식단 인식 1단계 | YOLOv8 — 음식 이미지 필터링 |
| 식단 인식 2단계 | Gemini Vision API — 음식 종류·섭취량 분석 |
| 영양 DB | 공공데이터 식품영양성분 DB 매핑 |
| AI 코칭 | LangChain + Pinecone + GPT-4o (RAG 파이프라인) |
| 지도 | 카카오맵 API |
| 푸시 알림 | FCM |
 
**Vision AI 파이프라인 동작 방식**
 
사진 업로드 → YOLOv8(음식 여부 판별) → Gemini Vision API(영양 분석) → 공공데이터 DB 매핑 → 식단 기록 저장
YOLOv8으로 비음식 이미지를 사전 차단해 API 비용과 응답 속도를 최적화한다.
 
**RAG 파이프라인 동작 방식**
 
사용자 질의 → 임베딩 → Pinecone 시맨틱 검색(식단 이력·질환 정보·운동 로그) → GPT-4o 컨텍스트 주입 → 개인 맞춤 답변 생성
"오늘 식단 괜찮아?" → "최근 3일 탄수화물 목표 12% 초과. 저녁은 닭가슴살 샐러드를 추천합니다."

## 차별점

### 경쟁 제품 개요

**삼성헬스** — 범용 건강 관리 플랫폼, 만성질환 특화 부재

직접 사용해본 결과, 삼성헬스는 걸음 수·칼로리·수면 등 활동 데이터를 수집하고 시각화하는 데 집중되어 있다. 식단 기록은 직접 입력 방식이며, 기록된 데이터를 바탕으로 일부 개인화된 해석을 제공한다. 그러나 기능의 중심이 일반 건강 관리에 있어 고혈압·당뇨 위험군에 특화된 식이 지도나 질환 맥락의 코칭은 제공되지 않는다. 삼성 기기 최적화로 범용성도 제한된다.

**필라이즈** — AI 코치·전문가 Q&A 있으나, 오프라인 연결 없음

사진 기반 AI 식단 인식, AI 코치, 전문가 Q&A, CGM 혈당 연동 등 기능이 잘 갖춰져 있다. 건강검진 데이터 연동과 영양제 분석을 강점으로 하며, 만성질환 관련 기능도 일부 포함되어 있다. 다만 AI 코칭이 사용자의 누적 식단·운동·질환 이력 전체를 실시간으로 참조하는 구조인지는 공개된 정보만으로 확인하기 어렵다. 헬스장·트레이너 연동 등 오프라인 연결은 제공하지 않는다.

**밀리그램 / 인아웃** — 2030 취향 UI·소셜 강점, 질환 특화 코칭 부재

식단 기록, 커뮤니티, 챌린지, 위젯 등 다양한 기능을 제공하며 2030 사용자 취향의 UI와 활성화된 커뮤니티가 강점이다. 기록 편의성을 높이기 위한 여러 방식(사진 저장, 즐겨찾기, 빠른 입력 등)을 지원한다. 그러나 직접 사용해보면 제공되는 기능이 많아 만성질환 관리에 집중하려는 사용자에게는 복잡하게 느껴질 수 있다. 질환 특화 코칭이나 개인 건강 이력 기반의 맥락적 조언은 제공되지 않는다.

### 핵심 차별성

**① 사진 한 장으로 영양 분석까지 자동 완결되는 식단 기록**

기존 앱들도 사진 저장, 검색, 즐겨찾기 등 기록 편의성을 높이는 방식을 제공한다. 필라이즈는 사진 기반 AI 인식도 지원한다. 그러나 촬영 한 장으로 음식 종류 식별 → 섭취량 추정 → 칼로리·영양소 자동 산출까지 한 번에 완결되는 2단계 파이프라인 구조는 On-Care의 차별점이다. YOLOv8로 음식 여부를 먼저 필터링해 불필요한 API 호출을 차단하고, 공공데이터 식품영양성분 DB 매핑으로 한국 음식 정확도를 확보한다. 인터뷰에서 확인된 기록 이탈의 핵심 원인("기록 노력 대비 피드백 부재")을 이 구조로 해결한다.

**② 개인 누적 이력을 실시간 참조하는 AI 코칭**

삼성헬스는 활동 데이터 기반의 개인화 해석을, 필라이즈는 AI 코치와 전문가 Q&A를 제공한다. 그러나 사용자의 이번 주 식단 기록·운동 이력·질환 정보를 동시에 참조해 "지금 이 질문"에 맥락 있는 답변을 생성하는 구조인지는 공개된 정보로 확인되지 않는다. On-Care는 LangChain + Pinecone + GPT-4o RAG 파이프라인으로 이를 구현한다. "오늘 식단 괜찮아?"라는 질문에 "이번 주 탄수화물 12% 초과, 저녁 닭가슴살 샐러드 추천"처럼 내 누적 기록 기준의 답변을 생성하는 것이 핵심이다. 인터뷰에서 확인된 "동일 식품 반복 섭취 시 대체 정보 필요", "혈당·나트륨 관련 맥락적 경고 부재" 니즈도 이 구조로 충족한다.

**③ 오프라인 운동 환경 연결**

삼성헬스·필라이즈·밀리그램·인아웃 모두 헬스장·트레이너 연동 기능이 없어 사용자가 별도 채널로 이탈해야 한다. On-Care는 카카오맵 API 기반 헬스장 검색·예약·트레이너 인앱 채팅을 통합하고, 트레이너 연결 시 사용자의 누적 건강 데이터 요약본을 자동 전달해 초기 상담의 반복 설명을 없앤다.

**④ 고혈압·당뇨 위험군 특화 설계**

삼성헬스는 범용 건강 관리, 밀리그램·인아웃은 체중 감량·소셜 중심이다. 필라이즈는 혈당 연동 등 만성질환 관련 기능을 일부 포함하고 있으나, 서비스 전체의 설계 방향이 다이어트·영양제 중심이다. On-Care는 서비스 기획 단계부터 타겟을 2030 고혈압·당뇨 위험군으로 명확히 설정하였다. 인터뷰에서 확인된 바, 외래 치료 수준의 당뇨 환자는 "혈당 급등·나트륨 과다 식품 회피"를 중심으로 식단을 관리하며 이를 지원하는 앱이 없다고 지적했다. 나트륨 섭취 경고, GI(혈당지수) 기반 식품 분류, 야식·불규칙 식사 패턴 감지를 On-Care의 핵심 기능으로 설계한 근거다.

## PMF (Product-Market Fit) 근거

국민건강보험공단 데이터 기준 20~30대 만성질환 유병률이 5년간 당뇨 38%, 고혈압 28% 등 증가하는 구조적 수요가 존재한다. 삼성헬스는 범용 건강 관리 중심으로 만성질환 특화 기능이 부재하고, 필라이즈는 혈당 연동 등 일부 만성질환 기능을 포함하나 오프라인 연결이 없다. 밀리그램·인아웃은 소셜·체중 감량 중심으로 질환 맥락의 코칭을 제공하지 않는다. 실제 환자 인터뷰를 통해 기존 앱이 실질적 행동 변화를 유도하지 못한다는 점이 실증되었으며, 이는 "기록 자동화 + 누적 이력 기반 맞춤 코칭 + 만성질환 특화 설계"라는 On-Care의 설계 방향이 충족되지 않은 시장 공백임을 뒷받침한다.

**인터뷰 결과 요약:**
 
- "헬스케어 앱을 써본 적 있냐"는 질문에 3인 모두 "써봤지만 3일 이상 지속한 적 없다"고 답함.
- 포기 이유: "매끼 입력이 너무 번거로워서" (3인 공통), "어차피 뻔한 조언밖에 안 해줘서" (2인)
- "사진을 찍으면 자동으로 기록되는 앱이 있다면 쓰겠냐"는 질문에 3인 모두 긍정적 반응.
- "내 이번 주 식단 기록을 기반으로 AI가 코칭해준다면"이라는 가설에 "그건 진짜 유용할 것 같다"는 반응 (3인 중 2인).

## ❓ On-Care 프로젝트 핵심 Q&A 정리

### 1. 서비스 기획 및 타겟

**Q: 왜 2030 세대의 만성질환 위험군을 타겟으로 했나요?**

**A:** 최근 5년간 2030세대의 당뇨병(38%)과 고혈압(28%) 유병률이 급증하고 있습니다. 하지만 이들은 불규칙한 생활 패턴으로 인해 기존 헬스케어 앱의 수동 기록 방식에 쉽게 피로를 느끼고 이탈하는 경향이 있어, 이를 자동화로 해결하고자 했습니다.


**Q: 기존 헬스케어 앱(삼성헬스, 인바디 등)과 무엇이 다른가요?**

**A:** 기존 앱이 체중 감량이나 체성분 측정에 집중한다면, On-Care는 **고혈압·당뇨 관리**에 특화되어 있습니다. 칼로리 계산을 넘어 나트륨, GI 지수 기반 분석을 제공하며, 기록이 단순 저장을 넘어 실질적인 오프라인 행동(헬스장/트레이너)으로 연결되는 구조를 가집니다.


### 2. 기술 스택 및 AI 아키텍처

**Q: YOLOv8과 Gemini Vision API를 함께 사용하는 이유는 무엇인가요?**

**A:** **비용 효율성**과 **분석 정밀도**를 동시에 잡기 위함입니다. 가벼운 YOLOv8이 음식 이미지 여부를 1차로 판별하여 불필요한 API 호출을 막고 , 음식인 경우에만 고성능 Gemini Vision API를 호출해 정밀한 영양 분석을 수행함으로써 비용을 획기적으로 절감합니다.


**Q: RAG(검색 증강 생성) 파이프라인에서 Pinecone을 선택한 이유는 무엇인가요?**

**A:** 완전 관리형 서비스로 인프라 운영 부담을 줄이면서도 밀리초 단위의 고속 검색을 지원하기 때문입니다. 또한 사용자별 메타데이터 필터링을 통해 개인의 식단·운동 이력에 기반한 **초개인화된 AI 코칭**을 구현하는 데 적합합니다.



### 3. 데이터 신뢰성 및 안전성

**Q: 사진 한 장으로 분석하는 영양 정보의 정확도를 어떻게 보장하나요?**

**A:** AI의 추정치를 그대로 쓰지 않고, 인식된 음식명을 **공공데이터 식품영양성분 DB**와 매핑하여 검증된 수치를 제공합니다. 또한 사용자가 직접 수정할 수 있는 UI를 제공해 데이터 무결성을 보완합니다.


**Q: AI가 잘못된 의료 조언을 할 위험은 어떻게 방지하나요?**

**A:** On-Care는 의료 행위가 아닌 '건강 관리 지원'을 목적으로 합니다. 시스템 프롬프트와 출력 필터를 통해 진단, 처방, 투약 관련 답변을 차단하며 , 신뢰할 수 있는 공공 의료 지침 데이터(한국영양학회 등)만을 소스로 사용하는 Closed-domain RAG 방식을 적용합니다.


### 4. 사용자 경험(UX) 및 행동 변화

**Q: 사용자가 앱을 꾸준히 쓰게 만드는 메커니즘이 있나요?**

**A:** 기록의 번거로움을 **Vision AI 자동화**로 낮추고 , RAG를 통해 "내 기록을 아는 맥락 있는 피드백"을 제공해 효능감을 높입니다. 또한 단순 접속이 아닌 실제 저나트륨 식단 달성이나 운동 완료 시에만 보상을 주는 **Streak 시스템**으로 건강 습관 형성을 유도합니다.


**Q: 오프라인 헬스장 연동은 구체적으로 어떻게 이루어지나요?**

**A:** 사용자의 위치를 기반으로 주변 헬스장을 검색해주며 , 사용자가 동의할 경우 앱에 축적된 식단 및 운동 요약 데이터를 트레이너에게 전달하여 초기 상담의 효율성을 극대화합니다.


### 5. 보안

**Q: 민감한 건강 데이터 보안은 어떻게 처리하나요?**

**A:** 개인 식별 정보(PII)를 제외한 익명화된 벡터 형태로 저장하며, 모든 데이터에 **AES-256 암호화** 및 **TLS 1.3** 전송 보안을 적용합니다. Pinecone 내에서도 user_id 기반 네임스페이스 분리로 데이터 격리를 보장합니다.
